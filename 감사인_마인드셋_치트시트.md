---

## 핵심 사고 전환표

| # | 개발자 본능 (❌ 끊어라) | 감사인 마인드 (✅ 장착해라) |
|---|---|---|
| 1 | 막자! (통제 구현) | **맞나?** (정확성 확인) |
| 2 | 기술적 최선을 고르자 | **비즈니스 목적**에 맞는 걸 고르자 |
| 3 | 문제 발견 → 내가 고친다 | 문제 발견 → **경영진에 권고**한다 |
| 4 | 돌아가면 됐지 (결과) | **절차를 밟았나?** (과정) |
| 5 | 효율이 우선 | **독립성/직무분리**가 우선 |

---

## PRIMARY/MOST/FIRST 보이면 이 순서

```
① 맞아?      (정확성, 목적 부합)
② 근거 있어?  (정책, 문서, 승인)
③ 지키고 있어? (접근 제한, 통제)
④ 확인했어?   (모니터링, 감사 추적)
```

**→ 개발자는 ③부터 생각한다. ①부터 생각해라.**

---

## 절대 규칙 5가지

**1. 쓰레기를 금고에 넣어봤자 쓰레기다**
- 값이 틀리면 접근 제한/감사 추적 다 의미 없음
- 보호할 가치가 있는지 먼저 확인

**2. 감사인은 경찰이 아니라 검사관이다**
- 직접 막는 사람 ❌ → "증거 보여줘" 하는 사람 ✅
- 구현/수정 선택지 → 오답일 확률 높음

**3. 기술이 있는지가 아니라 "왜" 있는지**
- 방화벽 있음 → "좋아" ❌
- 방화벽 있음 → "정책 근거가 뭐야?" ✅

**4. 비즈니스가 항상 기술 위에 있다**
- 최고 보안 기술 vs 비용 대비 적절한 수준 → 후자가 정답
- IT는 비즈니스를 지원하는 도구일 뿐

**5. 만든 사람이 확인하면 안 된다**
- 직무분리는 타협 불가
- "내가 만들었으니 내가 테스트" → 절대 안 됨

## 문제 풀 때 자가 점검

```
답 고르기 전에 3초만 멈추고 자문하기:

  "지금 만드는 사람으로 생각하고 있나?"
  → YES → 답 바꿔라
  → NO  → 그대로 가라
```

## 자주 걸리는 함정 패턴

| 함정 | 개발자 반응 | 정답 |
|---|---|---|
| 파라미터 검토 시 PRIMARY concern | 접근 제한 (D) | 값이 맞게 설정됐나 (A) |
| 로그인 지연 발견 | 대역폭 늘리자 (B) | 원인 파악 먼저 (D) |
| 서버 최적 구성 확인 | 벤치마크 (A) | 실제 활용률 (D) |
| 문제 발견 시 FIRST | 바로 조치 | 원인 파악 먼저 |
| 시간 부족 시 | 다 하려고 함 | 우선순위 정해서 중요한 것부터 |
| BCP 발동 기준 | 원인 파악 후 | **지속 시간(Duration)**이 트리거 |
| 이중화 (같은 건물) | "이중화 됐네" | 단일 장애점 = 함정 |
| 금융/은행 Reciprocal | 협력 좋네 | 기밀/경쟁/규제 → 거의 항상 오답 |
| "더 포괄적" 보기 | 완벽해 보임 | 오버엔지니어링 = 오답 |

## BCP 단계별 MOST important (헷갈리지 마라)

| 단계 | 키워드 | 정답 방향 |
|---|---|---|
| 개발(develop) | planning, designing | **현업 부서 참여** |
| 승인 | approval | 경영진 승인 |
| 구현(implement) | "after developed" | **인력에 전달(communicate)** |
| 유지 | up to date | Walk-through |
| 테스트 | verify, test | **한계(limitations) 식별** |

**→ "어느 단계"인지 먼저 확인. "이미 개발됨" 조건이면 승인 답은 탈락.**

---

## BCP 우선순위 규칙 (절대)

```
1. 인명 ★★★   ← "MOST important" 보기에 있으면 무조건 정답
2. 자산
3. 운영 연속성
4. 손실 최소화
```

**인명 보기 = 묻지 말고 그거. 백업/사이트/보험은 사람 다음.**

---

## 테스트 사다리 (점프 금지)

```
Paper → Walk-through → Tabletop → Functional → Parallel → Full Interruption
```

- 한 칸씩만 올라가라
- 이미 검증된 영역은 재검증 금지 (효율성)
- Test(결함 찾기) ≠ Training(사람 가르치기)

## 복구 지표 한 줄 정리

```
RTO = 시계 (얼마나 빨리 복구?)
RPO = 데이터 (얼마나 잃어도 돼?)
SDO = 대체 운영 시 서비스 수준
MTO/AIW = 최대 허용 중단 시간
```

---

## 답 바꾸기 절대 규칙

```
첫 직감 + 불안         → 그대로 가라
첫 직감 + "더 완벽해 보임" → 그대로 가라 (함정)
첫 직감 + 결정적 새 근거 → 바꿔도 됨
```

**"더 포괄적"은 CISA에서 오답일 확률이 높다. 정답은 최대가 아니라 최적.**

---

## 질문 동사 정확히 읽기

```
MOST / BEST         → 가장 좋은 것
PRIMARY             → 주된 목적 (부수 효과 X)
FIRST               → 시간 순서상 먼저
PARTIAL assures     → 부분만 보장 (방향 반전 주의)
LEAST               → 가장 약한 것
```

**PARTIAL을 MOST로 읽으면 정답이 정반대로 뒤집힌다.**

---

## 결함 ↔ 해결책 1:1 매칭 원칙

> 질문이 "X가 빠졌다" → 답은 "X를 명시하라"

관련 있는 Y가 아니라 **X 그 자체**가 정답. "역할/자원/절차"는 한 단계 멀다.

---

## Risk Assessment vs Gap Analysis

```
Risk Assessment → 지도 그리기 (탐색, 비즈니스 이해 ★)
Gap Analysis    → 지도 보고 빈 곳 찾기 (점검, 기준 필요)
```

**순서: Risk Assessment 먼저, Gap Analysis 나중.**
**"비즈니스 이해" 키워드 → Risk Assessment/BIA**
