## S1. CISA 자주 나오는 패턴 10가지 ★

| # | 패턴 | 핵심 |
|---|------|------|
| 1 | 근본이 정답 | Business Case > 에스크로, 파라미터 정확성 > 접근 제한 |
| 2 | 포괄적인 것이 정답 | 인증 > 무결성만, 변경관리 전체 > 일부 통제 |
| 3 | 시간/자원 부족 → 우선순위 | 중요한 것부터 |
| 4 | 순서 문제 → 먼저 하는 것 | 이해 먼저 → 평가 나중, 전체 그림 먼저 |
| 5 | 시스템 교체 → 데이터 마이그레이션 | 교체 최대 리스크 = 데이터 이관 오류 |
| 6 | 외부 노출 + 미검증 → 무단 접근 | 인터넷 노출 + 검증 안 됨 = Unauthorized Access |
| 7 | 패키지 SW → 파라미터 설정 | 기본 계정/비밀번호 비활성화 필수 |
| 8 | 원인 파악 먼저 | 문제 발견 → 바로 조치 ❌ → 원인 파악 ✅ |
| 9 | 예방 > 탐지 | 입력 시점 차단 > 출력 후 확인 (단, "detect" 물으면 탐지가 정답) |
| 10 | 대규모 → 단계별 배포 | 빅뱅 = 리스크 높음 |

## S2. Prevent vs Detect 통제 구분 ★★

```
같은 "무단 변경"이지만 동사가 다르면 답이 정반대!
  "prevent하려면?" → 접근 제한, 직무분리, 승인
  "detect하려면?"  → 로그 검토, 해시 비교, 모니터링
→ 문제 읽을 때 동사에 동그라미!
```

| 키워드 | 통제 유형 |
|--------|-----------|
| prevent, avoid, 막다 | 예방 통제 |
| detect, identify, 탐지 | 탐지 통제 |
| undetected, 탐지되지 않은 | 탐지 통제 |
| recover, restore | 복구 통제 |

**로그 함정**: "logged" = 통제 아님. "reviewed" = 통제.

## S3. 무단 변경 3가지 패턴 ★

| 키워드 | 묻는 것 | 정답 |
|--------|---------|------|
| 일관되게 적용/준수 확인 | 평가 활동 | 준수 테스트 |
| detect/undetected | 변경 사실 발견 | 해시 키 비교 |
| 추적/BEST 무단 변경 탐지 | 사후 역추적 | 실제 변경 → 승인 기록 역추적 |

```
순방향(문서→프로그램): 무단 변경 못 잡음 ❌
역방향(프로그램→문서): 모든 변경을 봄 → 무단 변경 발견 ✅
```

## S5. GREATEST/HIGHEST 풀이법 (CIA + 조직 맥락)

```
1단계: "이 시스템이 뭐 하는 거야?" → 비즈니스 목적
2단계: "CIA 중 뭐가 핵심?" → 우선순위
3단계: "그걸 위협하는 게 뭐?" → 정답
```

| 시스템 종류 | CIA 우선순위 |
|-------------|-------------|
| 공개 정보 사이트 | A (가용성) |
| 의료/금융 | C (기밀성) |
| 거래 데이터/회계 | I (무결성) |

## S6. 감사 테스트 순서 + 통제 4유형

```
1. 통제 설계 평가 → 2. 준수 테스트(Compliance) → 3. 실증 테스트(Substantive)
준수 실패 → 실증 X, 통제 결함 보고
```

| 유형 | 시점 | 예시 |
|------|------|------|
| **Preventive** (예방) | 사전 | 접근 제어, 직무분리, 입력 검증, 암호화 |
| **Detective** (탐지) | 사후 | 로그 검토, 모니터링, 해시 비교 |
| **Corrective** (시정) | 사후 | 복원, 롤백, 백업 복구, 패치 |
| **Compensating** (보상) | 대안 | 주 통제 어려울 때 보완 |

## S7. "FIRST" 문제 풀이법 ★

```
감사인 행동 순서: 이해 → 평가 → 권고 → 보고
FIRST = 거의 항상 "이해" (확인/검토/파악/인터뷰)

✅ 거의 정답: 확인 / 이해 / 검토 / 파악
❌ 거의 함정: 보고 / 권고 / 작성 / 변경
```

**예외**: 긴급 보안 사고(stolen/breached) → 즉시 봉쇄(Containment)

| # | 개발자 본능 (❌ 끊어라) | 감사인 마인드 (✅ 장착해라) |
|---|---|---|
| 1 | 막자! (통제 구현) | **맞나?** (정확성 확인) |

| 5 | 효율이 우선 | **독립성/직무분리**가 우선 |

## PRIMARY/MOST/FIRST 보이면 이 순서

```
① 맞아?      (정확성, 목적 부합)
② 근거 있어?  (정책, 문서, 승인)
③ 지키고 있어? (접근 제한, 통제)
④ 확인했어?   (모니터링, 감사 추적)
```

**→ 개발자는 ③부터 생각한다. ①부터 생각해라.**

## 절대 규칙 5가지

**1. 쓰레기를 금고에 넣어봤자 쓰레기다**
- 값이 틀리면 접근 제한/감사 추적 다 의미 없음
- 보호할 가치가 있는지 먼저 확인

**2. 감사인은 경찰이 아니라 검사관이다**
- 직접 막는 사람 ❌ → "증거 보여줘" 하는 사람 ✅
- 구현/수정 선택지 → 오답일 확률 높음

**3. 기술이 있는지가 아니라 "왜" 있는지**
- 방화벽 있음 → "좋아" ❌
- 방화벽 있음 → "정책 근거가 뭐야?" ✅

**4. 비즈니스가 항상 기술 위에 있다**
- 최고 보안 기술 vs 비용 대비 적절한 수준 → 후자가 정답
- IT는 비즈니스를 지원하는 도구일 뿐

**5. 만든 사람이 확인하면 안 된다**
- 직무분리는 타협 불가
- "내가 만들었으니 내가 테스트" → 절대 안 됨

## 자주 걸리는 함정 패턴

| 함정 | 개발자 반응 | 정답 |
|---|---|---|
| 파라미터 검토 시 PRIMARY concern | 접근 제한 (D) | 값이 맞게 설정됐나 (A) |

| 시간 부족 시 | 다 하려고 함 | 우선순위 정해서 중요한 것부터 |
| BCP 발동 기준 | 원인 파악 후 | **지속 시간(Duration)**이 트리거 |

| "더 포괄적" 보기 | 완벽해 보임 | 오버엔지니어링 = 오답 |

## DR 사이트 선택 규칙

| 사이트 | 복구 속도 | 비고 |
|---|---|---|
| **Hot** | 즉시 | 중요 애플리케이션 표준 |
| **Warm** | 수일 | 중요 앱엔 부적합 |
| **Cold** | 수주 | 비핵심용 |
| **Mirror/Replication** | 실시간 | RPO≈0 |
| **Reciprocal** | — | 금융/규제 산업 → 거의 항상 오답 |

**거리 규칙**: 복제 사이트는 **충분한 지리적 거리** 필수.
- 500m 같은 근거리 = 같은 재해(지진/홍수/정전)에 동시 피해 → 함정
- 일반 가이드: 핫사이트는 메인에서 충분히 떨어져 있어야 함 (수십~수백 km)

**판단 공식**: `사이트 종류(복구속도) × 거리(재해 격리) × 백업주기(RPO)` 균형이 좋은 것이 정답.
- 일일 백업 + 원거리 핫사이트 > 실시간 복제 + 근거리 핫사이트 (지역 재해 시)

## 클라우드 DR 책임 (감사인 관점)

> **"계약서에 없으면 벤더는 DR 의무 없다"**

경영진이 "벤더가 알아서 한다"고 하면 감사인의 NEXT step:

```
① 계약서(SLA) 검토 ★ ← 항상 출발점
   - DR 서비스 명시 여부
   - RTO/RPO 정의 여부
② 그 다음에야: 독립 감사보고서(SSAE 18 등) 검토
③ 그 다음에야: 벤더 DRP 사본 요청
④ 그 다음에야: 벤더 감사 계획
```

**핵심**: ②③④는 모두 ①(계약상 책임)이 확인된 후에야 의미가 있음.

## 통신 연속성(Telecom Continuity) 4종 구분

| 용어 | 핵심 키워드 | 한 줄 정의 |
|---|---|---|
| **Alternative routing** | 다른 **매체/네트워크** | 구리↔광, A통신사↔B통신사 — 매체 자체를 바꿈 |
| **Diverse routing** | **split/duplicate cable** | 같은 종류 케이블을 두 경로로. 같은 도관에 들어가면 한계 |
| **Long-haul diversity** | **장거리 캐리어** | 여러 장거리 통신사 패킷 스위칭 다중화 |
| **Last-mile protection** | **가입자 마지막 구간** | T-1, 마이크로웨이브, 동축 조합으로 로컬 루프 보호 |

**문제 키워드 → 정답 직행**:
- "split-cable / duplicate-cable" → **Diverse**
- "different medium / copper or fiber" → **Alternative**
- "long-distance carrier" → **Long-haul**
- "local loop / 가입자 구간" → **Last-mile**

**암기법**: **D**iverse = **D**uplicate cable (D끼리), **A**lternative = **A**lternate medium (A끼리)

## 전력 이상 유형과 보호 장치

### 전력 이상 유형

```
전압 높아지는 쪽 ↑
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
스파이크/서지  : 순간 급상승 (번개, 스위칭)
노이즈        : 잡음 섞임 (전자기 간섭)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
정상 전압 ───────────────────────
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
새그/딥       : 단기 전압 저하 (몇 초~몇 분)
브라운아웃     : 장기 전압 저하 (몇 분~몇 시간)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
전압 낮아지는 쪽 ↓

블랙아웃      : 완전 정전
```

### 보호 장치 1:1 매칭

| 전력 문제 | 보호 장치 | 하는 일 |
|-----------|-----------|---------|
| **급상승** (스파이크/서지) | **서지 보호기** | 과전압 차단/흡수 |
| **잡음** (노이즈) | **전력선 조절기** | 전압 평탄화 + 잡음 제거 |
| **단기 전압 저하** (새그) | **전력선 조절기** | 저장된 전력으로 골짜기 메움 |
| **단기 정전** (초~분) | **무정전 전원장치(UPS)** | 배터리로 버팀 |
| **장기 정전** (시간~) | **발전기** | 자체 전력 생산 |

### UPS 두 종류

```
온라인 UPS (이중 변환) : 항상 배터리 경유 → 전환 시간 0 → 핵심 시스템용
오프라인 UPS (대기)    : 평소 직결, 정전 시 전환 → 순간 끊김 → 일반 시스템용
```

### 정전 대응 순서 (시간축)

```
정전 발생 → UPS가 버팀(수분) → 발전기 가동(수시간) → 대체 사이트 이동(장기)
```

### 시험 함정

| 함정 | 왜 틀리나 |
|------|-----------|
| **중단 가능 전원장치**(Interruptible PS) | UPS가 아님. 이름이 비슷한 함정 |
| UPS로 장기 정전 대응 | UPS는 **단기**만. 장기 = 발전기 |
| 서지 보호기로 전압 저하 대응 | 서지 보호기 = 급상승만. 저하는 못 막음 |
| 발전기가 즉시 동작 | 시동 시간 필요 → 그 사이를 UPS가 버팀 |

### 한 줄 암기표

```
급상승 → 서지 보호기 | 전압 변동 → 전력선 조절기 | 단기 정전 → UPS | 장기 정전 → 발전기
```

## 답 바꾸기 절대 규칙

```
첫 직감 + 불안         → 그대로 가라
첫 직감 + "더 완벽해 보임" → 그대로 가라 (함정)
첫 직감 + 결정적 새 근거 → 바꿔도 됨
```

**"더 포괄적"은 CISA에서 오답일 확률이 높다. 정답은 최대가 아니라 최적.**

## 질문 동사 정확히 읽기

```
MOST / BEST         → 가장 좋은 것
PRIMARY             → 주된 목적 (부수 효과 X)
FIRST               → 시간 순서상 먼저
PARTIAL assures     → 부분만 보장 (방향 반전 주의)
LEAST               → 가장 약한 것
```

**PARTIAL을 MOST로 읽으면 정답이 정반대로 뒤집힌다.**

## 결함 ↔ 해결책 1:1 매칭 원칙

> 질문이 "X가 빠졌다" → 답은 "X를 명시하라"

관련 있는 Y가 아니라 **X 그 자체**가 정답. "역할/자원/절차"는 한 단계 멀다.

## Risk Assessment vs Gap Analysis

```
Risk Assessment → 지도 그리기 (탐색, 비즈니스 이해 ★)
Gap Analysis    → 지도 보고 빈 곳 찾기 (점검, 기준 필요)
```

**순서: Risk Assessment 먼저, Gap Analysis 나중.**
**"비즈니스 이해" 키워드 → Risk Assessment/BIA**
