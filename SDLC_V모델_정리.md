# SDLC & V모델 정리 (CISA 시험용)

---

## 1. SDLC (System Development Life Cycle)

시스템을 만들 때 따라야 하는 표준 절차/단계

```
1. 타당성 조사 (Feasibility Study)
2. 요구사항 정의 (Requirements Definition)
3. 설계 (Design)
4. 개발/코딩 (Development)
5. 테스트 (Testing)
6. 배포/구현 (Implementation)
7. 유지보수 (Maintenance)
```

---

## 2. V모델 전체 구조

```
        개발 (왼쪽)                          테스트 (오른쪽)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 요구사항 정의          ←─────→         인수 테스트 (UAT)
   "뭘 만들 것인가"                        "사용자가 원한 대로인가"

2. 시스템 설계            ←─────→         시스템 테스트
   "전체 구조를 어떻게"                     "전체 기능이 동작하는가"

3. 아키텍처/통합 설계     ←─────→         통합 테스트
   "모듈 간 연결을 어떻게"                  "모듈 간 연결이 되는가"

4. 상세 설계              ←─────→         단위 테스트
   "각 모듈 내부를 어떻게"                  "각 모듈이 동작하는가"

                    5. 코딩
                    실제로 만듦
```

**핵심: 각 테스트는 대응하는 개발 단계를 검증한다**

## 3. 개발 단계 상세 (위에서 아래로)

| 단계 | 하는 일 | 산출물 |
|---|---|---|
| 1. 요구사항 정의 | 사용자가 원하는 것 정리 | 요구사항 명세서, UAT 사양 |
| 2. 시스템 설계 | 전체 시스템 구조 설계 | 시스템 설계서 |
| 3. 통합 설계 | 모듈 간 연결 방법 설계 | 인터페이스 설계서 |
| 4. 상세 설계 | 각 모듈 내부 로직 설계 | 상세 설계서 |
| 5. 코딩 | 실제 개발 | 소스 코드 |

## 6. 테스트 유형별 정리

### 박스 테스트

| 테스트 | 코드 가시성 | 목적 |
|---|---|---|
| White Box | 코드 100% 확인 가능 | 모든 코드 경로 테스트 (코드 커버리지) |
| Gray Box | 일부만 확인 | 부분적 검증 |
| Black Box | 코드 안 봄 | 입출력만 확인 |

### 성능 테스트

| 테스트 | 목적 | 키워드 |
|---|---|---|
| Load Testing (부하) | 정상/예상 범위 내 성능 확인 | "75% 사용자만 로그인" → 부하 테스트 부족 |
| Stress Testing (스트레스) | 비정상/한계 초과 상황 확인 | "300% 사용자 몰림" |
| Volume Testing (볼륨) | 대량 데이터(레코드) 처리 확인 | 레코드 수 증가 |
| Recovery Testing (복구) | 장애 후 복구 능력 확인 | 시스템 복구 |

### 감사 테스트 기법

| 기법 | 하는 일 | 키워드 |
|---|---|---|
| Mapping | 실행된/안 된 코드 표시 | 코드 커버리지, 테스트 안 된 로직 찾기 |
| Snapshot | 트랜잭션의 로직 경로 흐름 기록 | 흐름 기록 |
| Tracing & Tagging | 실행된 명령어 추적 경로 | 실행 추적 |
| ITF (Integrated Test Facility) | 운영 환경에 가짜 엔티티로 테스트 | 별도 테스트 프로세스 불필요 |
| Test Data (시뮬레이션) | 미리 답 알고 입력 → 결과 비교 | 정확성 확인 |

## 10. QC vs QA

| | QC (품질 통제) | QA (품질 보증) |
|---|---|---|
| 시점 | 만든 후 | 만들기 전/중 |
| 방법 | 검사, 테스트, 감사 | 프로세스 설계, 표준, 가이드 |
| 목적 | 불량 발견 | 불량 예방 |
| 비유 | 치킨 맛보기 | 레시피 만들기 |
| 감사인 역할 | 내부 감사인 (본업) | 컨설턴트 (QA 개선 시) |

## 12. 자주 나오는 CISA 패턴

### 패턴 1: 근본이 정답
```
근본적인 것 > 파생적인 것
Business Case > 감사조항/에스크로
조직 영향 평가 > 포트폴리오 배치
프로젝트 책임 정의 > 문서화
```

### 패턴 2: 포괄적인 것이 정답
```
더 큰 개념 > 부분 개념
인증(출처+무결성) > 무결성만
핵심 통제 전체 > 직무분리만
```

### 패턴 3: 시간 부족 → 우선순위
```
시간/자원 부족 → 중요한 것부터 한다
기능 축소 파일럿 / 중요도 순 테스트
```

### 패턴 4: 순서 문제 → 먼저 하는 것이 정답
```
이해 먼저 → 평가는 나중
DFD 검토 먼저 → 통제 평가 나중
현재 상태 확인 먼저 → 미래 계획 나중
```

### 패턴 5: 시스템 교체 → 데이터 마이그레이션이 최대 리스크
```
시스템 전환/교체 문제 → 데이터 이관 오류가 가장 위험
```

### 패턴 6: 외부 노출 + 테스트 미완 → 무단 접근
```
인터넷에 노출된 컴포넌트 + 검증 안 됨 → Unauthorized Access
```

### 패턴 7: 패키지 소프트웨어 → 파라미터/설정이 핵심
```
패키지 구현 시 최대 리스크 = 파라미터 잘못 설정
벤더 기본 계정/비밀번호 비활성화 필수
```

### 패턴 8: 원인 파악 먼저 → 결론은 나중
```
문제 상황 발견 → 바로 결론/조치 ❌
문제 상황 발견 → 먼저 원인 파악 → 그다음 권고 ✅
예: 프로젝트 초과 → "관리 실패"라고 바로 보고 ❌ → 비즈니스 케이스/프로젝트 관리 검토 먼저 ✅
```

### 패턴 9: 예방 > 탐지
```
예방 통제(사전에 막기) > 탐지 통제(사후에 발견)
입력 오류 → 입력 시점에서 차단(Limit Check) > 출력 후 재계산
```

### 패턴 10: 크고 복잡한 시스템 → 단계별 배포
```
대규모 시스템 배포 = Phased Approach (단계별)
한 번에 배포(빅뱅) = 리스크 높음
```

## 13. 소프트웨어 품질 특성

| 특성 | 의미 | 비유 (자동차) |
|---|---|---|
| Functionality (기능성) | 필요한 기능이 존재하고 명세대로 동작하는가 | 핸들, 브레이크, 에어컨이 작동하는가 |
| Reliability (신뢰성) | 성능 수준을 유지할 수 있는가 | 10년 타도 고장 없는가 |
| Efficiency (효율성) | 성능 대비 자원 사용량은 적절한가 | 연비가 좋은가 |
| Portability (이식성) | 다른 환경으로 옮길 수 있는가 | 한국/미국 도로 다 달릴 수 있는가 |

## 14. CMM (Capability Maturity Model, 능력 성숙도 모델)

**조직의 소프트웨어 개발 프로세스 성숙도를 측정하는 모델**

```
Level 1: Initial (초기)        → 프로세스 없음, 개인 역량에 의존
Level 2: Repeatable (반복)     → 기본 프로세스 있음, 반복 가능
Level 3: Defined (정의)        → 조직 전체에 표준 프로세스 정의됨
Level 4: Managed (관리)        → 프로세스를 측정/관리
Level 5: Optimizing (최적화)   → 지속적 개선
```

**CMM이 보는 것:** "예측 가능한 프로세스를 따르는가?"
**CMM이 안 보는 것:** 제품 품질, 프로그래밍 효율, 보안 요구사항

## 15. 입출력 통제

### 입력 통제 (Input Controls)

| 통제 | 하는 일 | 예시 |
|---|---|---|
| Limit Check | 입력값이 정해진 범위 내인지 확인 | 시급: 10,000~100,000원 |
| Validity Check | 유효한 값인지 확인 | 부서코드가 존재하는 코드인지 |
| Completeness Check | 필수 항목이 다 입력됐는지 | 이름, 주소 빈칸 없는지 |

### 출력 통제 관련

| 통제 | 하는 일 | 시점 |
|---|---|---|
| Recalculation (재계산) | 수동으로 다시 계산해서 확인 | 출력 후 |
| Run-to-run Totals | 처리 단계별 합계 비교 | 출력 후 |
| Reconciliation (대사) | 파일 합계 대조 | 출력 후 |

**핵심:** 입력 오류가 원인 → 입력 통제(예방)가 BEST

## 16. 시스템 통합 시 통제

```
시스템A → 데이터 전송 → 시스템B

보내는 쪽 (출력 통제): "보내는 데이터가 정확한가?"
받는 쪽 (입력 통제):   "받은 데이터가 정확한가?"

→ 둘 다 검토해야 완전한 통제
인터페이스(통로)만 확인하면 부족
```

## 17. 기업 간 데이터 교환 시 보안

```
기업 간 데이터 교환 → 인증(Authentication)이 가장 중요

인증 = 출처 확인 + 무결성 확인
무결성 = 무결성만

인증 ⊃ 무결성 (인증이 무결성을 포함하는 더 큰 개념)
```

## 18. Transaction Journal (거래 저널)

**시스템에 입력된 모든 작업을 시간순으로 기록한 로그 (Git log와 유사)**

```
용도: 승인된 원본 문서와 대조 → 무단 입력 탐지

승인된 원본 문서 = "이 작업을 해도 된다"는 정식 승인 증거
  예: 구매요청서(팀장 승인), 급여변경 신청서(인사팀장 승인)

저널에는 있는데 원본 문서에 없다 → 무단 입력 발견!
```

## 19. 타당성 조사 (Feasibility Study) 범위

```
타당성 조사 = "할 수 있나? 비용은?" 큰 그림 수준

포함: 비용/리스크에 직접 영향을 주는 것 (예: 통신 프로토콜, 하드웨어 비용)
미포함: 세부 사항 (예: 암호화 알고리즘, 상세 통제 절차, 제3자 계약)

= "할까 말까" 판단에 필요한 것만 포함
```

## 20. 구현 후 검토 (Post-Implementation Review)

**배포 완료 후 "계획대로 됐는가?" 확인하는 단계**

```
하는 것:
  - ROI 분석: 비즈니스 케이스의 효과가 실현됐는가?
  - 요구사항 충족 확인
  - 통제 작동 확인

안 하는 것 (이미 끝났어야 할 것):
  - UAT → 배포 전에 했어야 함
  - 감사 추적 활성화 → 배포 시점에 했어야 함
```

## 21. 애자일 (Agile) 방법론 특징

```
핵심 원칙:
  1. 매 Sprint마다 "작동하는 코드" 산출 (데모용/미완성 ❌)
  2. 문서화보다 동작하는 소프트웨어 중시 (문서 최소화 정상)
  3. 매 Sprint 후 재계획 (정상)
  4. 팀원이 자율적으로 자원 관리 (PM이 아닌 팀원, 정상)

감사인 우려: Sprint에서 미완성/데모용 코드가 나오는 것
  → 다음 Sprint의 기반이 불안정해짐
```

## 22. Out-of-Scope (범위 외)

```
목적: 프로젝트의 경계(boundaries)를 명확히 하는 것
  = "이 프로젝트에서 안 하는 것" 목록
  → Scope Creep (범위 확장) 방지

비공식 목표 ❌ → 하든 안 하든 명확해야 함
유연성 제공 ❌ → 변경은 공식 변경 요청 프로세스로
```

---

## 23. 공모 (Collusion)

```
여러 직원이 합작하여 통제를 우회하는 것
= 잘 설계된 통제(직무분리 등)도 무력화할 수 있음
= 애플리케이션 시스템 통제 효과성에 가장 큰 위협
```

## 24. CI/CD (Continuous Integration / Continuous Deployment)

```
목적 (MAIN benefit): 소프트웨어 생성/배포 속도 향상

핵심 통제:
  - 모든 변경은 문서화 필수 (표준 유지보수 포함)
  - 최소 정보: 변경 설명, 성공 기준, 리스크, 롤백 절차
  - 개발자는 변경 로그에 쓰기 권한 없음 (올바른 직무분리)

MOST concern: 표준 유지보수 변경이 문서화되지 않는 것
```

## 25. 네트워크 보안/관리

### 네트워크 장비 구분

| 장비 | 역할 | 보안 목적? |
|---|---|---|
| Firewall (방화벽) | 허가/비허가 트래픽 판단하여 차단 | ✅ 보안이 주 목적 |
| Router | 패킷 경로 설정 (필터링은 부차적) | ❌ 라우팅이 주 목적 |
| Layer 2 Switch | MAC 주소 기반 트래픽 분리 | ❌ 허가/비허가 판단 안 함 |
| VLAN | 논리적 트래픽 분리 | ❌ 분리만, 보안 판단 안 함 |

**핵심:** "무단 트래픽 방지" = Firewall

### 네트워크 진단 도구

| 도구 | 하는 일 |
|---|---|
| Protocol Analyzer (Wireshark) | 패킷 캡처하여 네트워크 정보 모니터링/기록 (진단 도구) |
| Online Monitor | 통신 전송 정확성/완전성 측정 |
| Downtime Report | 통신 회선 가용성 추적 (보고서) |
| Help Desk Report | 운영 중 문제 기록 (보고서) |

### 네트워크 관리의 핵심

```
Configuration & Change Management (구성 및 변경 관리)
= 네트워크가 어떻게 동작하는지 설정하고 변경을 체계적으로 관리
= 기본 비밀번호 제거, 불필요한 서비스 비활성화 (Hardening)
= 네트워크 관리에서 가장 중요한 구성 요소
```

## 26. 서버/OS 보안

### Hardening (강화)

**시스템의 공격 표면을 최소화하는 모든 작업**

```
1. 최신 보안 패치 설치
2. 불필요한 서비스/포트 제거
3. 기본 계정/비밀번호 변경
4. 접근 권한 최소화
5. 안전하지 않은 프로토콜 비활성화
6. 불필요한 소프트웨어 제거
```

**OS 보안 설정 검토 = 파라미터 설정 확인** (OS도 패키지 소프트웨어)

## 27. 구성 관리 / 변경 관리 / CMDB

```
CMDB (Configuration Management DB):
  "현재 뭐가 있고, 어떻게 설정되어 있나?" (저장소)

Configuration Management (구성 관리):
  "설정을 체계적으로 관리한다" (관리 활동)

Change Management (변경 관리):
  "설정을 바꿀 때 승인 절차를 거친다" (변경 절차)

흐름:
  1. CMDB에 현재 상태 기록
  2. 변경 필요 → 변경 요청서 → 승인
  3. 변경 수행 → CMDB 업데이트

적용 범위: 서버, 네트워크, DB, 개발, 클라우드 등 IT 전체
```

## 28. 직무분리 (Separation of Duties, SoD)

```
개발자 → 개발 환경만 접근

UAT 후 운영 환경에서 코드 변경됨 → SoD 위반
= 개발자가 운영 환경에 접근 가능했다는 뜻
→ SoD 정책으로 해결
```

## 30. 데이터 처리 오류 탐지

### Hash Total (해시 합계)

```
처리 전후 합계를 비교하여 오류를 탐지하는 방법

처리 전: 직원번호 합계 = 158,432
처리 후: 직원번호 합계 = 158,432 → 일치 ✅
처리 후: 직원번호 합계 = 157,200 → 불일치 ❌ 오류 탐지!

※ 합계 자체는 의미 없지만 비교용으로 사용 → "해시" 합계
```

### detect vs prevent 구분

```
"detect(탐지)" 문제 → 탐지 통제 선택 (Hash Total, 로깅 등)
"prevent(예방)" 문제 → 예방 통제 선택 (Limit Check, 직무분리 등)
문제의 동사를 잘 읽어야 함!
```

## 35. 재해복구 사이트

### 사이트 유형

| 유형 | 상태 | 복구 시간 | 비용 |
|---|---|---|---|
| Hot Site | 즉시 사용 가능 (장비+데이터) | 가장 빠름 | 가장 높음 |
| Warm Site | 장비는 있지만 데이터 없음 | 중간 | 중간 |
| Cold Site | 빈 공간만 있음 | 가장 느림 | 가장 낮음 |

### 계약 시 핵심 조항

```
동시에 사이트를 사용할 수 있는 구독자 수
= 대체 사이트는 여러 기업이 공유
→ 재해 시 동시에 몰리면 자리가 없을 수 있음
→ 계약에 동시 사용 수/우선순위 명시 필수
```

### RTO vs RPO

```
RTO (Recovery Time Objective) = 복구 목표 시간
  "장애 후 얼마 안에 복구해야 하는가?"
  RTO 낮음 = 빨리 복구 = 비용 높음

RPO (Recovery Point Objective) = 복구 목표 시점
  "얼마만큼의 데이터 손실을 허용하는가?"
  RPO 낮음 = 데이터 손실 최소 = 비용 높음

둘 다 낮을수록 → 비용 증가
```

## 37. 용량/성능 모니터링

```
모니터링의 PRIMARY 목적:
  IT 자원 용량에 대한 정확한 피드백 제공
  = 현재 상태를 정확히 파악하는 것이 기본

IT 관리자가 용량을 모니터링하는 PRIMARY 이유:
  서비스 수준(SLA) 요구사항 충족 보장
  = IT의 궁극적 목적은 비즈니스 서비스 지원

구조:
  SLA 충족 (목적)
    └─ 정확한 모니터링 (기반)
        ├─ 최적 용량 운영 (수단)
        ├─ 미래 용량 예측 (수단)
        └─ 하드웨어 구매 판단 (수단)
```
