## D3-1. SDLC와 V모델

### SDLC (시스템 개발 생명주기)
```
1. 타당성 조사 (Feasibility Study)
2. 요구사항 정의
3. 설계
4. 개발/코딩
5. 테스트
6. 배포/구현
7. 유지보수
```

### V모델 전체 구조
```
        개발 (왼쪽)                          테스트 (오른쪽)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 요구사항 정의          ←─────→         인수 테스트 (UAT)
2. 시스템 설계            ←─────→         시스템 테스트
3. 아키텍처/통합 설계     ←─────→         통합 테스트
4. 상세 설계              ←─────→         단위 테스트
                    5. 코딩
```
**핵심: 각 테스트는 대응하는 개발 단계를 검증한다**

### 개발 단계 산출물
| 단계 | 산출물 |
|---|---|
| 요구사항 정의 | 요구사항 명세서, UAT 사양 |
| 시스템 설계 | 시스템 설계서 |
| 통합 설계 | 인터페이스 설계서 |
| 상세 설계 | 상세 설계서 |
| 코딩 | 소스 코드 |

### 테스트 단계
| 단계 | 누가 | 검증 대상 |
|---|---|---|
| 단위 테스트 | 개발자 | 상세 설계 |
| 통합 테스트 | 개발팀 | 아키텍처/통합 설계 |
| 시스템 테스트 | QA팀 | 시스템 설계 |
| 인수 테스트(UAT) | 사용자 | 요구사항 |

## D3-2. 개발 방법론 비교

### 방법론 종류
| 방법론 | 특징 | 주의점 |
|---|---|---|
| **Waterfall** | 순차적, 문서 중심 | 요구사항 안정 + 환경 안정 + 변경 가능성 낮을 때 적합 |
| **Prototyping** | 반복적, 빠른 수정 | 장점=시간·비용 절감 / 단점=변경 통제 어려움, scope creep |
| **Agile** | 짧은 스프린트, 점진적 | 매 Sprint마다 동작하는 코드 산출 (데모/미완성 X) |
| **RAD** | 빠른 개발, 도구 활용 | 소규모 적합. 자원 부족 시 FIRST=계획/접근법 검토 |
| **Spiral** | 반복 + 위험 분석 | 대규모·고위험 프로젝트에 적합 |

### 방법론별 핵심 포인트
```
OOD(객체지향) 핵심 장점 = 모듈 재사용 (성능/통제 향상 X)
CBD(컴포넌트 기반) 주요 장점 = 다중 개발 환경 지원
Prototyping 유일한 장점 = 시간·비용 절감
  단점: 내부 통제 약화, 변경 통제 복잡, scope creep, 미완성 운영 위험

Waterfall 적합 조건 = 요구사항 안정 + 환경 안정 + 변경 가능성 낮음
불확실성 높으면 → 반복적/Agile 방법론
```

## D3-3. 테스트 (유형/감사기법/문서)

### 박스 테스트
| 테스트 | 코드 가시성 | 목적 |
|---|---|---|
| White Box | 100% | 모든 코드 경로 (커버리지) |
| Gray Box | 일부 | 부분 검증 |
| Black Box | 안 봄 | 입출력만 |

### 성능 테스트
| 테스트 | 목적 | 키워드 |
|---|---|---|
| Load (부하) | 정상/예상 범위 성능 | "75% 사용자만 로그인" → 부하 부족 |
| Stress | 비정상/한계 초과 | "300% 사용자 몰림" |
| Volume | 대량 데이터 처리 | 레코드 수 증가 |
| Recovery | 장애 후 복구 능력 | 시스템 복구 |

### 감사 테스트 기법
| 기법 | 하는 일 | 키워드 |
|---|---|---|
| Mapping | 실행된/안 된 코드 표시 | 코드 커버리지 |
| Snapshot | 트랜잭션 로직 경로 흐름 | 흐름 기록 |
| Tracing & Tagging | 실행 명령어 추적 | 실행 추적 |
| ITF | 운영 환경에 가짜 엔티티로 테스트 | 별도 프로세스 불필요 |
| Test Data | 미리 답 알고 입력 → 결과 비교 | 정확성 |

### 추가 테스트 유형
| 테스트 | 목적 | 키워드 |
|---|---|---|
| Sandbox | 격리/가상 환경에서 실행 | 안전한 테스트 |
| Compatibility/Interface | 시스템 간 연동 | 호환성 |
| Regression | 변경 후 새 오류 없는지 확인 | 기존 기능 보호 |
| Sociability | 기존 시스템에 악영향 없이 공존 | "coexist" |
| Parallel | 신·구 시스템 동시 실행, 결과 비교 | 사용자 요구사항 충족 확인 |
| Alpha | Beta 직전, 내부(개발자) 테스트 | 내부 |
| Beta | 최종 단계, 외부 사용자(UAT 형태) | 외부 |

### 테스트 접근법
```
Top-down = 인터페이스 오류 조기 발견
Bottom-up = 핵심 모듈 오류 조기 발견

테스트 순서: Unit → Integration → System → UAT → Stress/Performance
실패 영향: Unit(최소) < Integration < System < Acceptance(최대) — 늦을수록 영향 ↑
시간 부족 → UAT는 절대 생략 불가
```

### 테스트 데이터와 환경
```
테스트 데이터 = 질 > 양 (유효+무효, 모든 조건 대표)
운영 데이터(Sanitized) 장점=실제 대표 / 단점=모든 기능 커버 불가
운영 데이터 테스트 사용 = 경영진 사전 승인 필수
테스트 환경+운영 데이터 = 기밀성 리스크 (개발자 특권 접근)
테스트 데이터 = 삭제 X → 회귀 테스트용 보존

Race Condition/Heisenbug:
  비운영 환경+계측 도구 → 타이밍 변경 → Race Condition 놓침
```

### 결함 감소 비용 효율
```
가장 비용 효율적 = 공식 소프트웨어 검사(Inspection)
  → 조기 발견 → 전파 방지 → 재작업 최소
```

### 계산 정확성 검증
```
계산 정확성 검증 = 시뮬레이션 거래(Test Deck) → 사전 결정 결과와 비교
  소스 코드 검토 X
```

### SW 품질 지표
```
SW 품질 평가 = 최초 보고 MTBF (순수 결함)
  MTTR·응답 시간 = 대응팀 성과 (품질 평가 아님)

시간 제한 테스트 → 중요도+사용 빈도 기준 리스크 기반 우선순위
  자동화 = 효율성 향상이지 커버리지 보장 아님

QA+UAT 결합 최대 리스크 = **부적절한 수용**
  사용자 만족 시 QA 표준 미달도 수용 압력 → 목적이 다른 테스트는 분리 필수
```

### CGI / 인터넷 접점 리스크
```
CGI 스크립트 QC 미검증 최대 리스크 = 비인가 접근
  공개 인터넷 서버에서 실행 → 내부 침투 경로
```

### 테스트 문서 시점
| 문서 | 시점 |
|---|---|
| UAT 사양 | 요구사항 정의 단계 |
| 상세 테스트 계획 | 시스템 테스트 단계 |
| 테스트 데이터 | 시스템 테스트 단계 |
| QA 테스트 사양 | 개발 후반 |

## D3-4. 프로젝트 관리 (도구/역할/문서)

### PM 도구
| 도구 | 용도 | 키워드 |
|---|---|---|
| Gantt Chart | 일정 대비 진행 비교 | 진행 모니터링 |
| PERT | 3가지 시간 추정으로 임계경로 | 계획 단계 |
| Critical Path | 최단 완료 시간 | Slack=0, Crashing |
| Function Point | 개발 규모 추정 | 모듈/속성 기반 |

### 핵심 역할 구분
| 역할 | 키워드 |
|---|---|
| User Management | deliverables 승인, sign-off |
| Steering Committee | 전체 감독, 마일스톤 |

| Project Manager | 일상 관리 |
| QA Team | 표준 준수 |

|---|---|---|

| Feasibility Study | 할 수 있는가 | 타당성 미검증 |
| Quality Plan | 품질 관리 방법 | 품질 통제 불가 |

### 프로젝트 추정 도구
| 도구 | 목적 |
|---|---|
| **PERT** | 기간 추정 (3점: 최악/최선/정상) |
| **COCOMO** | 개발 비용 추정 |
| **Earned Value** | 비용 vs 일정 통합 추적 (비용 단독 평가 금지!) |

```
EVA: Schedule Variance = EV - PV (음수=지연)
인력 추가 시 = 크리티컬 패스 먼저 재검증 (경로 변경 가능)
Crashing = Slack 0 활동(Critical Path)에 프리미엄 투입 → 조기 완료

계획 이탈 식별 = Project Performance Criteria (Framework/Approach/Resource Plan X)
SDLC 단계/산출물 결정 = 초기 계획 단계 (추적·자원 관리 위해, 리스크 전부 식별까지 기다릴 수 없음)
QA/품질 관리 = 프로젝트 전반 지속적 적용 → 결함 조기 발견 → 비용 최소화
Initiation 회의 감사 최우선 = 복잡성·리스크 분석 (자원 결정은 복잡성에 종속)
```

### Scope Management
```
기능 초과 + 예산 초과 = Scope Management 실패
Scope Creep = 범위가 통제 없이 확대되는 현상
Baseline 부적절 → Scope Creep → 시간/예산 초과

Stop/Freezing Point = 변경 금지 X → 이후 변경의 비용 효과성 평가 의무화
증상(예산 초과)이 아닌 근본 원인(범위 확대)을 찾아라

Out-of-scope 섹션 목적 = 프로젝트 경계(Boundaries) 명확화
  유연성은 변경 요청 프로세스로 관리
```

### Project / Program / Portfolio
```
Project = 단일 프로젝트
Program = 관련 프로젝트 그룹
Portfolio = 전체 프로젝트 총괄

"복수 프로젝트 관리/통제 효과성" → Portfolio
포트폴리오 최우선 = 비즈니스 플랜과 정렬
```

### 프로젝트 성공/실패 요인
```
IS 프로젝트 실패 #1 원인 = 요구사항 단계에서 사용자 참여 부족
  요구 변경은 "증상", 참여 부족이 "원인"
구현 성공 최대 요인 = 전반적인 조직 환경 (기술/방법론보다 상위)

일정 OK+예산 OK+대량 비계획 초과 근무 = 계획 비현실적 징후 → 계획 정확성 조사
프로젝트 초기에 역할·책임 공식 정의 필수 — 초기 단계 오류 = 후속 단계 비용 최대

비즈니스 케이스 유효성 상실 → 업데이트 먼저 → 시정 조치 식별 → 그 후 중단/계속 판단
표준 위반 원인=시간/리소스 부족 → 리소스 추가로 해결 (지연/처벌/사후 수정 X)
Post-project Review = 교훈(Lessons Learned) | Post-implementation Review = 통제·목표 달성 검증
다단계 구현 1단계 실패 → 감사 초점 = 1단계 변경이 나머지에 미치는 영향 (통제 검증보다 우선)
리엔지니어링 이슈 식별 → Gap Analysis (As-Is vs To-Be)
일정 달성 검증 = 완료 작업 패키지 기반 외삽(Extrapolation) — 직접 관찰 > 보고서 > 인터뷰 (80:20 편향 주의)
```

### 리스크 관리
```
프로젝트 리스크 = 시작 전/초기에 식별 ("아직 이르다" = 틀린 주장)
리스크 관리 = 선제적(forward-looking), 사후 대응 = 실패
Initiation 회의 감사 최우선 = 복잡성·리스크 분석
```

## D3-5. QC vs QA / 품질 / CMM

### QC vs QA
| | QC (품질 통제) | QA (품질 보증) |
|---|---|---|
| 시점 | 만든 후 | 만들기 전/중 |
| 방법 | 검사, 테스트 | 프로세스 설계, 표준 |
| 목적 | 불량 발견 | 불량 예방 |
| 비유 | 치킨 맛보기 | 레시피 만들기 |
| 감사인 역할 | 내부 감사인(본업) | 컨설턴트 |

### 소프트웨어 품질 특성
| 특성 | 비유 (자동차) |
|---|---|
| Functionality | 핸들/브레이크/에어컨 작동 |
| Reliability | 10년 타도 고장 없음 |
| Efficiency | 연비 좋음 |
| Portability | 한국/미국 도로 다 달림 |

### CMM (능력 성숙도 모델)
```
1. Initial      → 프로세스 없음, 개인 역량 의존
2. Repeatable   → 기본 프로세스, 반복 가능
3. Defined      → 조직 전체 표준 프로세스
4. Managed      → 측정/관리
5. Optimizing   → 지속적 개선
```
- CMM이 보는 것: 예측 가능한 프로세스를 따르는가
- CMM이 안 보는 것: 제품 품질, 프로그래밍 효율, 보안 요구사항

## D3-6. 입출력 통제와 시스템 통합

### 입력 통제
| 통제 | 예시 |
|---|---|
| Limit Check | 시급 10,000~100,000원 |
| Validity Check | 부서코드 존재 여부 |
| Completeness Check | 필수 항목 누락 여부 |

### 출력 통제 (사후)
| 통제 | 시점 |
|---|---|
| Recalculation | 출력 후 수동 재계산 |
| Run-to-run Totals | 단계별 합계 비교 |
| Reconciliation | 파일 합계 대조 |

**핵심:** 입력 오류가 원인 → 입력 통제(예방)가 BEST

### Hash Total (탐지)
```
처리 전: 직원번호 합계 158,432
처리 후: 158,432 → 일치 ✅ / 157,200 → 오류 탐지 ❌
※ 합계 자체는 의미 없지만 비교용 → "해시" 합계
```

### 시스템 통합 시 통제
```
시스템A → 데이터 → 시스템B
보내는 쪽(출력 통제) + 받는 쪽(입력 통제) 둘 다 검토
인터페이스(통로)만 보면 부족
```

### 추가 입력 통제
```
Check Digit = 전치(transposition)·전사(transcription) 오류 탐지에 효과적
  Range=범위, Validity=유효값, Duplicate=중복 → 전치/전사 탐지 불가
원격 데이터 검증 = 전송 전 원격 사이트에서 (소스에 가깝게)
입력 오류→출력 오류 시 = Limit Check(입력 예방 통제) 권고
```

### Override 통제
```
Override 통제 = 자동 로깅 + 독립적 검토 (직무 분리)
보상 통제를 "있을 것"이라고 가정 금지
```

### Automated Systems Balancing
```
트랜잭션 손실 방지 = 입력 총합 vs 출력 총합 자동 비교
Checksum = 무결성(Integrity) 보장 (변경 탐지). 인증/부인방지는 디지털 서명
```

### 공모(Collusion) = 통제 효과성 최대 위협
```
여러 직원 합작 → 통제 우회
잘 설계된 직무분리도 무력화 가능
= 애플리케이션 시스템 통제 효과성에 가장 큰 위협
```

### 기업 간 데이터 교환
```
인증(Authentication)이 가장 중요
인증 = 출처 확인 + 무결성 → 인증 ⊃ 무결성 (포괄적, 패턴 2)
```

### Transaction Journal (거래 저널)
```
시스템 입력 작업의 시간순 로그 (Git log와 유사)
용도: 승인된 원본 문서와 대조 → 무단 입력 탐지
저널엔 있는데 원본 문서 없음 → 무단 입력!
```

## D3-8. 시스템 전환 전략

| 전략 | 방식 | 위험 | 비용 |
|---|---|---|---|
| **Direct Cutover** | 한 번에 교체 | 최고 | 최저 |
| **Pilot** | 일부 사이트 먼저 | 중간 | 중간 |
| **Phased** | 모듈별 단계적 | 중간 | 중간 |

```
테스트 시간 부족 → 기능 축소 파일럿 릴리스 (개발팀 테스트 생략 X)
```

## D3-9. 구현/배포

### Post-Implementation Review (PIR)
```
목적: 설계대로 작동? 비즈니스 요구사항 충족?
  = 만능: 설계 검증 + 접근 통제 + 가치/ROI 측정
  주목적 = 비즈니스 요구사항 충족 확인

구현 후 감사 최우선 = 접근 통제(Access Control) 설정 확인
```

### Certification vs Accreditation
```
Certification = 통제 준수 평가
Accreditation = 운영 승인 (경영진 리스크 수용)
```

### 구현 시 주의사항
```
ERP/ERM = 벤더 패키지 → 상세 설계 문서 검토 불필요
촉박한 일정에 기능 추가 금물 → Future Enhancement로
Code Freeze 후에는 테스트만 수행
벤더 SW 구현 핵심 = 기본 계정/비밀번호 비활성화
패키지 SW 구현 최대 리스크 = 파라미터 설정 오류
```

### 데이터 마이그레이션
```
마이그레이션 승인 = Data Owner (DBA/PM/감사인 X)
시스템 교체 최대 리스크 = 데이터 마이그레이션 오류
마이그레이션 최우선 = Semantic(의미적) 일치 (Arithmetic보다 상위)
분산 시스템 데이터 무결성 감사 FIRST = DFD 검토
```

## D3-10. 소프트웨어 획득 (Acquisition)

```
획득 감사 핵심 = 계약서가 법률 자문(Legal Counsel) 검토/승인 받았는지
Acquisition ≠ Development → 감사 포인트가 다름

Make vs Buy 결정:
  최우선 판단 기준 = 조직 내 기술 역량(core competency)
  Make: 내부 역량 충분 + 핵심 비즈니스
  Buy: 역량 부족 + 비핵심 + 검증 솔루션 존재

획득 검토 핵심 = 제품이 현재/계획 OS와 호환되는지
오프쇼어 개발 성공 핵심 = 상세 명세서 (의사소통 갭)
외부 계약 시 품질 계획(Quality Plan) = 반드시 포함
```

## D3-11. IS 감사인 역할 (개발 프로젝트)

```
감사인 = 발견 + 권고 (직접 해결/구현 = 독립성 위반)
독립성 유지 = 검토(Review)/평가 ✅
독립성 훼손 = 자문(Advise)/권고/추정 ⚠ (구현 참여)

통제 사양 평가 = 요구사항 단계 (가장 빠른 시점에 참여)
절차 이탈 발견 → 먼저 승인 여부 확인 (기술 확인은 그 후)

BPR 시 감사인 최우선 = 핵심 통제 적용 확인 (SoD는 통제 중 하나일 뿐, 전체 통제 체계가 우선)
BPR 완료 후 감사 초점 = Post-BPR 프로세스 흐름도 (현재 상태, 과거 X)
프로젝트 초과 → 원인 파악 먼저(비즈니스 케이스+PM 검토) — 성급한 결론·교체 권고 금지

통합 감사(Integrated Audit) = 자원 최적화 + 품질 개선 동시 달성 (여러 감사 유형 동시 수행·결과 공유)
감사인 본업 = QC(감사·검토) / QA 개선 참여 시 = 컨설턴트 역할
시스템 타 부서 확장 시 최우선 = 프로세스 소유자 식별 (모니터링·승인 통제 책임 주체)
```

## D3-18. 기타 (타당성/PIR/Agile/CI-CD/EUC)

### 타당성 조사 범위
```
"할까 말까" 판단에 필요한 것만
포함: 비용/리스크 직접 영향 (통신 프로토콜, HW 비용)
미포함: 세부(암호화 알고리즘, 상세 통제, 제3자 계약)
```

### 구현 후 검토 (PIR, Post-Implementation Review)
```
하는 것: ROI 분석, 요구사항 충족, 통제 작동 확인
안 하는 것 (이미 끝났어야): UAT, 감사 추적 활성화
```

### Agile
```
1. 매 Sprint마다 작동하는 코드 산출 (데모/미완성 ❌)
2. 문서화보다 동작 SW (문서 최소화 정상)
3. Sprint 후 재계획 (정상)
4. 팀원 자율 자원 관리 (정상)

감사인 우려: 미완성/데모용 코드 → 다음 Sprint 기반 불안정
```

### Out-of-Scope
```
프로젝트 경계 명확화 → Scope Creep 방지
"안 하는 것" 명시
비공식 목표 ❌, 유연성 제공 ❌ → 변경은 공식 절차
```

### CI/CD
```
MAIN benefit: 생성/배포 속도 향상
핵심 통제:
  - 모든 변경 문서화 필수 (표준 유지보수 포함)
  - 최소 정보: 변경 설명, 성공 기준, 리스크, 롤백
  - 개발자는 변경 로그 쓰기 권한 없음 (직무분리)
MOST concern: 표준 유지보수 변경 미문서화
```

### EUC (End-User Computing)
```
사용자 직접 개발 (Excel 매크로, Access DB)
장점: 비용/시간 절감
최대 리스크: 테스트/IT 일반 통제 대상이 아닐 수 있음
  → 코드 리뷰 ❌, 테스트 ❌, 변경 관리 ❌
  → 그런데 경영진이 정식 시스템처럼 의존
```

## D3-19. CISA 풀이 팁 (Domain 3)

```
"BEST" = 가장 포괄적인 답
"MOST important" = 절대 생략 불가한 것
"initial/first" = 가장 기본적인 검증

획득/조달 감사 최우선 = 비즈니스 케이스 승인
비즈니스 케이스 평가 최우선 = ROI (TCO는 비용만, ROI는 비용 대비 혜택)
비즈니스 케이스 준비 최우선 = 타당성 입증(Feasibility) + 비용편익분석

타당성 조사 포함: 비용/리스크 영향 큰 항목 (통신 프로토콜 등)
타당성 조사 제외: 상세 절차 (암호화·통제·계약)
```
